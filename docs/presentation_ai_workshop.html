Why Multi-Agent?

Single-prompt approaches fail for domain reasoning

The problems with one-shot prompts

  • Context window limits -- clinical data + market data + buyer data exceeds useful context for a single call
  • Role confusion -- the model conflates "what is this drug" with "who should buy it" and produces muddled reasoning
  • No separation of concerns -- a single prompt cannot enforce that science outputs feed into market inputs
  • Math contamination -- LLMs hallucinate arithmetic when mixing reasoning with calculation

The multi-agent solution

  • Specialized agents with clear ownership boundaries
  • Each agent owns a section of shared state
  • Deterministic code handles all math
  • Pipeline topology enforces data dependency order
"One agent, one concern, one section of state."

Architecture Overview

Sequential pipeline -- not parallel -- because of data dependencies

Ownership Split Principle

Each agent owns exactly one concern -- no overlap

AgentOwnsRule
ScienceAsset propertiesMOA, clinical data, PTRS, regulatory signals
MarketCompetitive environmentSOC comparator, differentiation, peak sales
SynthesizerBuyer-side reasoningWho buys, urgency, bidding, deal price
"If it is about the asset itself, science owns it.
If it is about the market, market owns it.
If it is about who buys it, synthesizer owns it."

Why this matters

  • Each prompt stays focused -- shorter context, better reasoning
  • Agents cannot contradict each other within their domain
  • Testable: you can unit-test science output independently of market
  • Editable: swap the market agent without touching science or synthesis

Anti-pattern avoided

  • Science agent prompt explicitly says: "Do NOT analyze market competition or buyers"
  • Market agent prompt says: "Do NOT reason about potential buyers"

Key design decisions

  • total=False -- fields are optional, populated incrementally
  • Each agent reads fields from prior agents, writes only its own
  • Top-level BDState holds cross-agent outputs: buyers, bidding_tension, three scenario_* dicts

Implementation

# backend/utils/ptrs_lookup.py
def get_adjusted_ptrs(phase, ta, signals):
    base = lookup_ptrs(phase, ta)  # from JSON table

    # 50plus and 100plus are mutually exclusive
    has_100 = "patients_dosed_100plus" in signals

    total_adj = 0.0
    breakdown = []
    for signal in signals:
        if signal == "patients_dosed_50plus" \
           and has_100:
            continue  # skip lower tier
        contrib = _DERISKING_ADJUSTMENTS[signal]
        breakdown.append({
            "signal": signal,
            "contribution": contrib
        })
        total_adj += contrib

    adjusted = min(base + total_adj, _PTRS_CAPS[phase])
    return {"base": base,
            "adjusted": adjusted,
            "breakdown": breakdown}

Design decision: additive not multiplicative

  • Easier to reason about: each signal adds a fixed percentage
  • Easier to tune: domain experts adjust one number per signal
  • Phase caps prevent runaway PTRS (no Phase 1 asset at 80%)

The new_class_creation Verdict

Special case -- when SOC is symptom-only and the asset is disease-modifying

When does it trigger?

  • Current SOC addresses symptoms rather than underlying pathology
  • Analyst / KOL language includes "first-in-class", "disease-modifying", "new standard of care"
  • The asset targets a pathway where no approved drug exists

Category expansion math

# Instead of displacement from existing SOC:
value = (ta_category_sales
         * expansion_multiplier  # 1.5-2.0x
         * 0.35)  # capture rate

# Rubric from the market prompt:
# conservative: 1.5x expansion
# base case:    2.0x expansion
# capture rate: 35% of expanded category

Example: narcolepsy (Centessa / Xywav)

  • Current SOC (Xywav): manages symptoms -- sodium oxybate for cataplexy and EDS
  • If a new asset is orexin receptor agonist (addresses underlying orexin deficiency), it is disease-modifying
  • Cannot size on Xywav displacement alone -- the market expands because a true treatment brings in patients who refused symptom management
"You cannot size a disease-modifying drug by displacement from a symptom-management SOC. The market category itself grows."

Surfaced as

"Category creation scenario" -- distinct from displacement, communicated separately in the three-scenario output.

Three concerns

Defensive buyer detection

If a company's drug IS the SOC comparator identified by the market agent, they are flagged as a defensive buyer -- they may pay a premium to prevent disruption.

Deal Multiple Approach

Why NPV formulas fail for pharma M&A

The PTRS-over-discounting problem

  • Standard DCF: discount future cash flows by risk rate
  • In pharma: PTRS already captures binary approval risk
  • Applying both PTRS and a high discount rate double-penalizes the asset
  • Result: DCF consistently undervalues early-stage assets vs actual deal prices

Solution: deal multiples

deal_multiple =
    base(phase)
  + urgency_adj
  + bidding_adj
  + differentiation_adj

# Strategic value:
if_success = peak_displacement * deal_multiple
risk_adj   = if_success * weighted_avg_ptrs

Dual-View Design Pattern

Risk-adjusted vs if-succeed -- always show both

Why two values?

  • Risk-adjusted (PTRS applied): for investment committee decisioning -- "what is the expected value?"
  • If-succeed (assumes approval): for negotiation ceilings -- "what is the max we would pay?"
  • The gap IS the risk: a GP sees at a glance how much value depends on clinical success

Example output

ScenarioIf-SucceedRisk-Adj
Standalone$2.0B$0.6B
Displacement$8.7B$2.6B
Strategic$7.0B$2.1B

The 3x gap in strategic scenario signals high clinical risk (Phase 1/2, PTRS ~30%).

Computed in parallel per indication

# backend/agents/synthesizer.py
# Pre-compute dual values before LLM call
for ind in indications:
    ly  = ind.get("launch_year")
    npv = _npv_discount(ly)
    ptrs = ind.get("ptrs_adjusted", 0.25)
    peak = ind["peak_sales_with_displacement_bn"]

    # If-succeed: no PTRS applied
    ind["peak_sales_if_succeed_bn"] = \
        round(peak * npv, 2)

    # Risk-adjusted: PTRS applied
    ind["peak_sales_risk_adjusted_bn"] = \
        round(peak * ptrs * npv, 2)
"Dual-view is more useful than a single estimate. Optimistic and pessimistic bounds bracket the negotiation."

Anti-Anchoring in Prompts

The problem with injecting conclusions into prompts

The anchoring problem

  • If you inject "$2B-$4.5B" into a prompt, the LLM will output a number within that range -- regardless of the actual evidence
  • This is cognitive anchoring, amplified by LLM sycophancy
  • Earlier versions of this system passed pre-computed deal ranges into the synthesizer and got suspiciously "reasonable" outputs

What went wrong

# BAD: anchoring the LLM
prompt = f"""Comparable deals suggest a
range of $2B-$4.5B for this asset.
What is your valuation?"""
# LLM: "$3.2B" -- always in range

The solution

# GOOD: provide facts, not conclusions
"""Comparable transactions:
  [M&A] Merck / AssetX
    TA: oncology | Phase: 2
    Deal: $1.8B upfront ($3.2B total)
  [M&A] Pfizer / AssetY
    TA: oncology | Phase: 3
    Deal: $5.1B upfront

Assess how similar/different the
current asset is to each comp,
then derive your own estimate --
do NOT simply copy a comp's value."""
"Provide data, not conclusions. Let the model reason from facts."

Implementation

deal_benchmarks.py returns raw deal records -- no pre-computed ranges, no averages, no suggested bands.

Prompt Engineering Patterns

Five patterns used across the agent system

1. Fixed vocabulary for structured extraction

"DE-RISKING SIGNAL VOCABULARY --
 use ONLY these exact strings:"
  orphan_drug, fast_track, ...
# LLM picks from the list, code validates

2. JSON schema in prompt with examples

"Return ONLY valid JSON (no markdown):"
{
  "indications": [{
    "name": "exact indication name",
    "science_score": 8.0,
    ...
  }]
}

3. Strict rubric with exact thresholds

# Differentiation verdicts
"best_in_class: 2+ efficacy metrics
 meaningfully better, no key worse"
"better_in_class: 1 metric better,
 none worse"

4. Separation: LLM reasons, code computes

LLM doesCode does
Identify de-risking signalsCompute PTRS score
Assess differentiation verdictSelect displacement %
Map buyer franchise fitLook up urgency multipliers
Reason about comparatorsNPV discount arithmetic

5. Parameters, not free-form

# revenue_multiplier as prompt parameter
"revenue_multiplier = 4.0 for standard"
"revenue_multiplier = 5.0 for orphan"
"revenue_multiplier = 3.0 for competitive"

# deal_multiple formula in prompt
"deal_multiple = base(phase)
  + urgency_adj + bidding_adj"

The LLM selects from defined options -- it does not invent its own multipliers.

Validation Methodology

Blind test against real deals -- the system must discover, not memorize

Blind test protocol

  • Feed the system only the asset name
  • No hardcoded comparators, no deal price hints
  • System must discover the SOC, classify differentiation, identify buyers, and price the deal
  • Compare output to actual announced deal price

Pass criterion

Strategic scenario (if-succeed) within 20-30% of actual deal price.

Why if-succeed for comparison?

Announced deal prices are the buyer's if-succeed bet -- they already applied their own risk framework internally.

The recalculate endpoint

Re-enter the graph mid-pipeline with user overrides:

# User adjusts PTRS or peak sales manually
# Re-run from synthesizer only

# 1. Take existing state snapshot
state = last_run_state.copy()

# 2. Apply user overrides
state["indications"][0]["ptrs_adjusted"] = 0.45

# 3. Re-enter at synthesizer node
result = synthesizer_node(state)

Pattern benefits

  • Incremental feedback: frontend shows progress per agent
  • Partial re-run: change one assumption, re-price without re-running science + market
  • Debuggability: inspect state after each node
  • Trust but verify: every assumption is editable by the user

Lessons Learned

Five principles from building a production multi-agent system

1. Domain experts define the rubric

The differentiation verdicts, de-risking signals, and deal multiple tables came from BD professionals -- not from asking the LLM to invent criteria. The model applies the rubric; it does not create it.

2. Hardcode slow, search fast

Patent cliff dates change quarterly. Deal velocity changes weekly. Hardcode the former, Tavily-search the latter. The buyer_context.py file is updated once per quarter; deal sentiment is searched live every run.

3. Never let LLMs do math

PTRS computation, NPV discounting, deal multiple arithmetic -- all in Python. The LLM identifies which signals apply and which verdict fits. Code turns those into numbers. Errors dropped to near-zero after this split.

4. Dual-view beats single estimate

Risk-adjusted for IC decisioning, if-succeed for negotiation ceilings. The gap IS the risk visualization. A 3x gap signals high clinical risk without needing a paragraph of explanation.

5. Every assumption should be editable

The recalculate endpoint lets users override PTRS, peak sales, urgency multipliers, or differentiation verdicts and re-run the synthesizer in seconds. Trust the system's defaults, but verify by adjusting. "Trust but verify" is the only way domain experts adopt AI tools.
